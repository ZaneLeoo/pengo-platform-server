# RuoYi 项目协作约定

## Java 代码规范

以下规则适用于本仓库全部 Java 模块，包括系统、框架、MES、Agent、定时任务和代码生成模块。

### 格式

- 使用 Java 常见的 K&R 大括号风格，类、方法和控制语句的左大括号必须与声明同行。
- 使用 4 个空格缩进，禁止 Tab；单行建议不超过 120 个字符。
- 禁止通配符 import，提交前执行 Spotless 检查。
- 后端不使用 `record`，DTO、实体和结果对象统一使用普通 Java 类。
- 在语义清晰且不影响框架兼容性的前提下，优先使用 Lombok 减少 Getter、Setter、构造器等样板代码。
- 数据库实体类的每个业务字段必须提供准确的字段级注释；状态、类型等字段还应说明枚举范围。

### 单一职责

- 一个类只承担一个可清晰命名的职责；出现“并且”“同时负责”时应考虑拆分。
- Controller 只负责 HTTP 参数、权限和响应，不拼装业务结果、不访问 Mapper、不实现业务规则。
- ToolService 只负责把业务服务结果适配为 Agent 工具协议，不承担持久化和核心业务计算。
- Application/Domain Service 负责业务用例和规则；Mapper 只负责数据访问。
- Dify SSE 解析、事件标准化、文件持久化、工具元数据等能力必须保持独立组件。
- 优先使用依赖注入和组合，避免静态全局状态与跨层调用。

### 类与方法规模

- 方法应只完成一个动作，建议不超过 40 行；复杂条件应提取为有业务含义的私有方法。
- 类建议不超过 300 行；超过时必须检查是否混合了协议、业务、存储或第三方适配职责。
- 方法参数过多时使用明确的请求对象，但不要创建无业务含义的万能 Map。
- 返回值优先使用强类型 DTO，禁止使用无明确结构的万能 Map；Agent 工具统一返回 `AgentToolResult<T>`。

### 硬编码与注释

- 稳定状态、动作和业务类型使用枚举或结果码类型，禁止散落字符串判断。
- 注释解释业务原因和边界，不复述代码。
- 对外方法和承载业务规则的方法编写方法级 Javadoc。

### 验证

- 格式化：`mvn spotless:apply`
- 格式检查：`mvn spotless:check`
- 模块测试：根据改动范围执行对应模块及其依赖模块测试。
- Agent 测试：`mvn -pl ruoyi-agent -am test`
- 后端编译：`mvn -pl ruoyi-admin -am -DskipTests compile`
